# Project Brief — Personal Recipe App
> Format: BMAD Method · For Claude Code

---

## 1. Product Vision

A **responsive web app** (mobile, tablet, desktop) for managing personal recipes. The experience focuses on visual discovery, frictionless recipe entry, and intuitive organization. AI plays an increasing role across versions to reduce manual input effort.

---

## 2. Tech Stack — Suggestions

The final stack decision is left to BMAD / architecture phase. The suggestions below are starting points, not constraints.

| Layer | Suggestion | Accepted alternatives |
|---|---|---|
| Frontend | React (SPA) | Next.js, Remix |
| Backend | Supabase (BaaS) — fast to set up | Node.js/Express, Firebase |
| Auth | Supabase Auth | Auth.js, Clerk |
| OCR / AI | Claude API (Anthropic) server-side | GPT-4o Vision, Google Vision AI |
| Hosting | Vercel + Supabase | Railway, Render, Fly.io |

> The project owner is **open to any technical recommendation** BMAD sees fit.

---

## 3. Features — Scope by Phase

### Phase 1 — MVP Core (V1)

**Fast recipe entry — one-page form**
- **Single-page form** (no multi-step wizard): title, ingredients + quantities, steps, photo, tags
- Goal: entry as fast as possible, zero friction
- Advanced fields (time, difficulty, calories) can be optional or auto-filled later

**Recipe management (CRUD)**
- Create, read, update, delete a recipe

**Ingredient system**
- Predefined shared ingredient catalog
- Ability to add custom personal ingredients
- Autocomplete on input

**Tags & Categories**
- Free tags assignable to each recipe
- Predefined system tags (e.g. quick, slow-cooked, comfort food, light, veggie…)

**Home page — Discovery view**
- **Thematic carousels** driven by tags (no plain list)
- Examples: "Quick to make", "Slow-cooked", "Comfort food", "Light / Low-cal"

**Authentication**
- Each user sees only their own recipes

---

### Phase 2 — Sharing (V2)

**Share recipe via link**
- Generate a unique public link per recipe
- Accessible without login, read-only

---

### Phase 3 — Smart photo import (V3)

**OCR / Recipe import**
- User uploads a photo or screenshot (cookbook, magazine, website)
- Server-side Claude API call → automatic extraction and structuring
- Result pre-fills the form → user reviews and confirms

---

### Phase 4 — Auto metadata (V4)

**Auto-calculated prep time & difficulty**
- Based on ingredients and steps, automatic estimation via AI or business rules

**Calorie calculator**
- Approximate calorie estimate based on ingredients and quantities
- Accuracy is not critical — a rough range is enough

---

### Phase 5 — Minimal input (vX)

**Full auto-metadata generation**
- User only enters: title + ingredients/quantities + instructions
- All tags and metadata (time, difficulty, calories, category) are **automatically generated by AI**
- Goal: zero categorization effort for the user

---

## 4. UX / UI Guidelines

- **Minimal and appetizing** design — large photos, readable typography
- Mobile-first; bottom nav bar on mobile, grid layout on desktop
- Recipe creation: **one-page form**, fields ordered for smooth entry
- Recipe reading mode: optimized for cooking (large fonts, step-by-step scroll)
- Carousels are horizontally scrollable on mobile

---

## 5. Audience & Context

- Very small circle: owner + spouse + a few close friends
- No elaborate onboarding needed — landing on an empty app is fine
- No high security requirements; main risk to mitigate is **bot abuse of APIs** (rate limiting, especially on AI endpoints)

---

## 6. Authentication

- **Goal**: avoid friction of classic email/password if possible
- **Preferred option**: simple **username-only login** (no password) — to be validated by BMAD based on security implications
- **Acceptable alternatives**: magic link email, Google/Apple OAuth (passwordless SSO)
- **Excluded**: password-based account creation in V1

> BMAD will decide on feasibility and risk based on the chosen stack.

---

## 7. Search

- **Basic search bar** from V1 (search by title, ingredient, tag)
- No advanced full-text search required to start

---

## 8. Internationalization

- **V1: French only**
- Whether to anticipate i18n from the start is a **decision left to BMAD** (trade-off between effort and future flexibility)
- An English version is planned for a future vX

---

## 9. Serving size adjustment

- **Not in V1**
- In vX: automatic ingredient quantity recalculation based on number of servings

---

## 10. Constraints & Assumptions

- No social features in V1 (no likes, comments, follows)
- No meal planning in V1
- All AI calls happen **server-side** (API keys never exposed to the client)
- Calorie calculation is intentionally approximate — not a medical tool

---

## 11. Open Questions (for BMAD architecture phase)

1. **Auth**: is username-only login feasible without significant security risk? If not, what passwordless alternative?
2. **Bot protection**: is rate limiting enough, or is a light auth layer needed on public endpoints?
3. **Backend**: Supabase or custom stack? What does BMAD recommend?
4. **i18n**: anticipate from V1 or refactor later?
5. **Image storage**: Supabase Storage, Cloudinary, or other?
6. **AI metadata**: synchronous call (slight latency acceptable) or background async task?
